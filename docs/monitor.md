# EMBLOCS Runtime Monitor

This document captures the design of the EMBLOCS runtime monitor: the
Python-based PC tool and the minimal C monitor program that runs on the
target. It covers the connect sequence, metadata strategy, data model
extensions, the read/write protocol, and design principles that govern
what belongs in the monitor versus what belongs in a block.

This document is a design record, not a finished specification. It will
be reorganized and merged into the main documentation at a future date.

---

## 1. Design Principles

### 1.1. Monitor as Infrastructure, Not Feature Delivery

The central design principle is: **if a capability can be implemented as
a block, it must be implemented as a block.** The monitor protocol is
necessary infrastructure, not a feature delivery mechanism. The only things
that belong in the monitor are things that cannot be done any other way.

Applying this criterion strictly, the monitor's irreducible responsibilities
are:

- The connect/metadata exchange — blocks cannot describe themselves to an
  external tool; that requires a protocol layer.
- Primitive memory read — a block cannot read its own address out to the PC.
- Primitive memory write — a block cannot receive an arbitrary value from
  the PC and write it to an arbitrary address.
- Binary packet passthrough — the monitor receives and forwards binary
  packets generated by blocks, but does not need to understand their
  contents.

Everything else is a candidate for a block.

### 1.2. The Block Alternative

Many capabilities that would require dedicated monitor protocol support in
other frameworks are naturally expressed as blocks in EMBLOCS:

- **Oscilloscope** — a block with N raw input pins that runs in a thread
  and emits binary packets containing sampled signal values.
- **Performance measurement** — a `perf_timer` block with `start` and
  `stop` functions and a `clocks` output pin. Place `perf_timer.start`
  before and `perf_timer.stop` after the code under test in a thread.
  No dedicated monitor support needed.
- **Signal injection** — a block that receives values from binary packets
  and drives output pins. The monitor delivers packets; the block
  interprets them.
- **Streamer/sampler** — blocks analogous to LinuxCNC HAL's streamer and
  sampler components. A streamer block receives binary packets from the
  monitor and writes values to output pins each thread execution. A sampler
  block reads input pins each thread execution and emits binary packets.
- **Network/bus interfaces** — blocks that transfer data between pins and
  external communication channels (a second UART, CAN bus, Ethernet, SPI,
  a custom PIO protocol). From EMBLOCS's perspective these are structurally
  identical to any other block.

The block approach has a key advantage over dedicated monitor features:
blocks pay no overhead unless they are wired into the system. A system
without a perf_timer block pays nothing for performance measurement
capability. A system without an oscilloscope block pays nothing for
oscilloscope capability.

### 1.3. The Two-Sided Architecture

The monitor is divided into two parts with a clean boundary:

- **PC side** — a Python tool that holds the complete system model, does
  all command parsing, owns the UI, and drives the protocol. Heavy logic
  lives here where it is easy to iterate.
- **Target side** — a minimal C program that executes read/write actions
  on target memory on request, transmits metadata on request, and passes
  through binary packet streams generated by blocks. Logic here is kept
  as thin as possible.

### 1.4. Feature Taxonomy

Blocks fall into three natural categories:

- **Computation blocks** — pure function, no hardware, no external
  communication. PID, summer, integrator, limit, mux. Most portable
  and reusable.
- **Hardware blocks** — interface directly to MCU peripherals. GPIO, ADC,
  PWM, UART, SPI, CAN. Platform-specific but structurally identical to
  computation blocks from the system designer's perspective.
- **Communication blocks** — bridge between the pin/signal world and an
  external data stream. Oscilloscope, streamer, sampler, network interface,
  monitor packet receiver. These are what make the system observable and
  injectable from outside.

The monitor protocol only needs to support the packet streams that
communication blocks generate and consume.

## 2. Object Model Extensions

The existing `emblocs.py` object model does not contain any target memory
addresses. Four additions are required to support runtime monitoring. All
new fields are `int | None`, initialized to `None` and populated during
the connect sequence.

### 2.1. `FieldDef.field_offset`

The byte offset of this struct field (pin pointer or pin pointer array)
within the block instance struct, as produced by `offsetof()` at compile
time. Stored as a `uint16_t` on the target; instance structs are always
under 65536 bytes.

For array fields, this is the offset of the first element. The address of
an individual pin pointer within the array is computed on the PC side:

```
pin_addr = base_addr + field_offset + (flat_index * sizeof(void*))
```

where `flat_index` for a 2D array pin `[i][j]` with inner dimension size
`N` is `i*N + j`, derivable from `PinDef.field_indices` and `FieldDef.dims`
without further target communication.

### 2.2. `FunctDef.func_addr`

The target address of the C function implementing this block function.
Needed for thread introspection and eventually for runtime function
reordering (item 6 in the capability list).

### 2.3. `BlockInstance.base_addr`

The target address of the statically allocated instance struct for this
block instance. Combined with `FieldDef.field_offset`, gives the absolute
address of any pin pointer within the instance.

For statically allocated instances — the current and near-term deployment
model — these addresses are fixed at link time and can be emitted as
constants in `system.c`.

### 2.4. `Signal.value_addr`

The target address of the 32-bit signal value. This is what meters and
the oscilloscope read. Distinct from pin pointer addresses — a signal
address is where the value lives, not where a pointer to it lives.

### 2.5. Connectivity Reconstruction

Signal-pin connectivity does not require separate metadata. Given
`BlockInstance.base_addr` and `FieldDef.field_offset` for each pin, the
monitor can compute the address of each pin pointer and read its value
from target memory. Comparing that value against the known
`Signal.value_addr` for each signal reconstructs the full connection graph.
This is one of the first operations performed after the address data is
populated.

Dummy signal addresses may eventually need to be tracked, but this is
deferred until there is a concrete need to read the value of an unconnected
pin directly from target memory.

## 3. Hashing Strategy

### 3.1. Who Computes the Hash

The Python toolchain computes all hashes at build time, over the in-memory
`Design` object. The hash values are emitted as C constants into `system.c`.
The target stores them as opaque constants and returns them on request
without knowing or caring what they represent.

The monitor computes the same hashes over the `Design` it has just built
(from disk or from target metadata) and compares to what the target returned.

### 3.2. Two Hashes

Two separate hashes are maintained:

- **Structural hash** — covers block instances, signal-pin links, and
  thread-function links. A mismatch means the wiring has changed; the
  monitor should abort or warn strongly.
- **Value hash** — covers initial signal values. A mismatch means only
  initial values have changed; the monitor can warn but probably continue.

This gives the monitor a richer mismatch report than a single combined hash.

### 3.3. Hash Algorithm

Deferred. CRC32 is the likely choice for simplicity; SHA-1 or MD5 provide
better collision resistance but are probably overkill.

### 3.4. `.blocs` Path on Target

The target also stores the path to the `.blocs` file as a short string
constant. This allows the monitor to locate the file autonomously at
connect time without user input. If the file is not found at the stored
path, the monitor falls back to asking the user.

## 4. Connect Sequence

### 4.1. Overview

Both metadata paths share the same skeleton. Only step 2 differs.

1. Monitor sends `HELLO`. Target responds with: protocol/format version,
   metadata mode (`FULL` or `HASH_ONLY`), structural hash, value hash,
   and `.blocs` file path.

2. **Path A (full metadata on target):** Monitor requests metadata from
   target in dependency order — BlockSpecs, BlockDefs, BlockInstances,
   Signals, Threads — and builds a `Design` object. See section 4.2.

   **Path B (hash only):** Monitor loads `.blocs` file from disk (using
   the stored path), builds `Design` via the existing Python pipeline.
   See section 4.3.

3. Monitor computes structural hash over the `Design` and compares to
   target's value. Abort or warn strongly on mismatch.

4. Monitor computes value hash over the `Design` and compares to target's
   value. Warn (but likely continue) on mismatch.

5. `GET_STRUCT_OFFSETS` — monitor requests `field_offset` arrays, one per
   BlockDef. Populates `FieldDef.field_offset` for all fields.

6. `GET_INSTANCE_ADDRS` — monitor requests `base_addr` for each
   BlockInstance. Populates `BlockInstance.base_addr`.

7. `GET_SIGNAL_ADDRS` — monitor requests `value_addr` for each Signal.
   Populates `Signal.value_addr`.

8. Monitor reads pin pointers from target memory and reconstructs the
   signal-pin connection graph. Verifies against the model built in step 2.

9. Monitor is ready to operate.

Steps 5–8 are identical regardless of which path was taken in step 2.

### 4.2. Path A: Metadata from Target

Metadata is requested in dependency order since reconstructing a `Design`
requires the same ordering as building one from a `.blocs` file:

- **BlockSpecs** — compressed `.bloc` file content, one per unique block
  type. See section 5.1.
- **BlockDefs** — name + BlockSpec identifier + param/value pairs + offsetof
  arrays. See section 5.2.
- **BlockInstances** — name + BlockDef identifier + base address.
  See section 5.3.
- **Signals** — name + type + value address. See section 5.4.
- **Threads** — name + period_ns + list head address. See section 5.5.

Each category is requested separately. The target responds with a count
followed by that many records.

### 4.3. Path B: Metadata from Disk

The monitor receives the `.blocs` path from the `HELLO` response, locates
and loads the file, and builds the `Design` using the existing
`blocs_parser.py` pipeline. This path is not purely "from disk" — the
`base_addr`, `field_offset`, and `value_addr` data still comes from the
target in steps 5–7. Path B is "structure from disk, addresses from
target."

Path B is appropriate for memory-constrained targets where storing full
metadata in flash is not feasible. Path A is preferred when flash permits,
because it does not depend on the availability of build artifacts on the PC.

### 4.4. A Third Path: From ELF File

A possible future path (not a current priority) would use `pyelftools`
to extract symbol addresses directly from the `.elf` file, bypassing the
need for the target to transmit address data. The trust problem (does this
ELF match the running binary?) can be addressed by comparing a hash of the
`.text` section against a constant stored in the firmware. This path is
noted here but deferred.

## 5. Target Metadata Representation

### 5.1. BlockSpec

A BlockSpec is stored as a compressed `.bloc` file. Compression discards:
- Comments (`//` and `///` annotations)
- Descriptions
- `var` declarations (private to the block implementation, not needed for
  monitoring)
- `include` declarations (not needed for monitoring)
- All whitespace

Keywords and common keyword sequences are replaced by an escape character
followed by a single-character code. Names and expressions are stored
unmodified. The result is an ASCII string that can be transmitted over the
UART text channel and expanded back through the existing `.bloc` parser.

This approach is efficient for blocks with array pins — it stores the name
template, not the expanded pin names. If multiple BlockDefs are derived
from one BlockSpec, the BlockSpec is stored only once.

### 5.2. BlockDef

A BlockDef is stored as: variant name + BlockSpec identifier (integer index
into the transmitted BlockSpec list) + parameter values in the order defined
by the BlockSpec. A more compact form would use only the parameter values
(since their order is defined by the BlockSpec), with an integer index
identifying the BlockSpec.

The `field_offset` arrays (one `uint16_t` per struct field, in declaration
order) are stored with the BlockDef metadata, since offsets are per-type
not per-instance.

### 5.3. BlockInstance

A BlockInstance is stored as: instance name + BlockDef identifier (index
or name) + base address of the instance struct.

### 5.4. Signal

A Signal is stored as: name + type + address of the 32-bit value in target
memory. Signal-pin connectivity is not stored explicitly — it is
reconstructed by reading pin pointers from target memory and comparing
against known signal addresses.

### 5.5. Thread

A Thread is stored as: name + period_ns + address of the head of the
execution list. The execution list is a linked list of function/data pointer
pairs. The monitor traverses it from target memory to reconstruct the
ordered function list.

## 6. Read/Write Protocol

### 6.1. Channel Assignment

Read and write commands use the **ASCII text channel**, not the binary
packet channel. Rationale:

- Read/write traffic is sporadic and does not need hard real-time delivery.
- The binary packet channel is reserved for time-sensitive streaming data
  (oscilloscope samples, sampler output) where latency must be minimal and
  predictable.
- A read or write command should be deferred if a binary packet needs to
  be transmitted; this deferral is natural on the ASCII channel and
  problematic on the binary channel.
- ASCII commands are human-readable, which significantly aids debugging
  during early development. A terminal session is sufficient to verify
  that the C monitor is functioning correctly.

### 6.2. UART Requirements

The C monitor will most likely run in a non-realtime background context
(idle task or background loop) while EMBLOCS threads run in interrupt
context or RTOS tasks. This creates receive buffer overflow risk: if a
thread blocks the monitor for an extended period, incoming command bytes
accumulate in the hardware UART buffer (typically 4-8 bytes) and overflow.

**Interrupt-driven UART receive with a software buffer is a baseline
requirement for reliable monitor operation.** A polled UART implementation
may work in practice if realtime load is low, but this is not guaranteed
and is the target author's responsibility to verify.

Recommended minimum software receive buffer: large enough to hold at least
one complete command with margin. Given command lengths of 12-19 bytes,
64 bytes is a reasonable minimum.

### 6.3. Request/Response Discipline

The protocol uses strict request/response: the PC sends one command and
waits for the complete response before sending the next. This is not an
unoptimized starting point — it is the correct baseline given the
asymmetric bandwidth constraint:

A read command is approximately 12 bytes; its response is approximately
23 bytes. The response is nearly twice as large as the command. At any
baud rate, the target can only produce responses at roughly half the rate
the PC can send commands. Pipelining commands would cause the target's
transmit queue to grow without bound.

Strict request/response naturally throttles the PC to the target's response
rate and works correctly regardless of UART buffer depth.

### 6.4. Bulk Read Command

For connect-time operations that require many reads (e.g., reading all pin
pointers to reconstruct the connection graph), a bulk read command reduces
the number of round trips:

**Command:** `R<addr>:<len>\n`

- `<addr>` — 8 hex digits, the starting address
- `<len>` — 2 hex digits, the number of 32-bit words to read (1-255)

**Response:** `R<addr>:<len>:<w0>,<w1>,...,<wN>\n`

- Each word is 8 hex digits
- Words are comma-separated
- The echoed address and length allow the PC to verify it received the
  correct response

Example — read 6 words starting at 0x20001A00:

```
Command:  R20001A00:06\n
Response: R20001A00:06:AABBCCDD,00000000,12345678,DEADBEEF,00FF00FF,CAFEBABE\n
```

Word count rather than byte count is used because all EMBLOCS values are
32-bit aligned; partial-word reads are never needed.

### 6.5. Single Word Write Command

Write operations are always single-word. Bulk writes are not supported
because: (a) they are not needed for the current capability set, and (b)
longer commands increase the risk of receive buffer overflow.

**Command:** `W<addr>:<value>\n`

- `<addr>` — 8 hex digits
- `<value>` — 8 hex digits

**Response:** `W<addr>:OK\n`

- The echoed address confirms which write completed

Example — write 0x00000001 to address 0x20001A04:

```
Command:  W20001A04:00000001\n
Response: W20001A04:OK\n
```

### 6.6. Baud Rate

Standard FTDI USB-serial cables commonly support 1 Mbaud to 3 Mbaud, which
provides significantly more throughput than 115.2 kbaud. Higher baud rates
reduce the transmission time component of round-trip latency and make
connect-time bulk reads faster. The protocol is baud-rate agnostic; baud
rate is a configuration parameter.

## 7. Binary Packet Channel

### 7.1. Overview

The binary packet channel uses COBS encoding embedded in the ASCII stream.
COBS-encoded packets never contain a zero byte; zero marks packet boundaries.
Packets are always less than 254 bytes, so COBS adds exactly one byte of
overhead per packet.

The monitor's role with respect to binary packets is purely as a conduit.
It receives packets, identifies their type, and dispatches them to the
appropriate handler. It does not generate binary packets itself.

### 7.2. Packet Types

Packet types are defined by the blocks that generate them, not by the
monitor protocol. A new communication block (oscilloscope, sampler, network
bridge) defines a new packet type. The monitor adds a handler for that
packet type. No other protocol changes are required.

### 7.3. Oscilloscope Packets

Two oscilloscope block variants are anticipated:

**Streaming variant** — suited to RAM-constrained targets. The block runs
in a thread, reads N raw input pins each execution, and emits one binary
packet per execution containing the N values. Ring buffer and trigger logic
reside on the PC side in the monitor. The monitor sees a continuous stream
of sample packets and applies trigger logic to decide when to display a
waveform.

**Ring buffer variant** — suited to targets with more RAM. The block
maintains a ring buffer in target memory. It samples N raw input pins into
the ring buffer each thread execution. When a trigger condition is met,
sampling stops and the block transmits the captured buffer as a sequence
of binary packets. All timing-critical work happens on the target; the
monitor simply receives and displays.

The monitor can determine how to interpret packet payloads from the Design
object: the oscilloscope block's input pins are connected to signals of
known types, so the monitor knows the type and label of each value in the
packet without additional metadata.

## 8. Capability Roadmap

The following is the prioritized list of runtime capabilities, with
implementation notes.

| # | Capability | Priority | Notes |
|---|-----------|----------|-------|
| 1 | Read signal/pin values (meter) | Must-have | Primitive read + Design model |
| 2 | Write signal/pin values (knob) | Must-have | Primitive write + Design model |
| 3 | Stream signal values (oscilloscope) | Must-have | Oscilloscope block + binary packet channel |
| 4 | Inject signal values (signal injector) | Nice to have | Streamer block variant |
| 5 | Reconnect pins and signals | Advanced | Requires pin/signal metadata with offsets |
| 6 | Reorder functions in thread | Low value | Requires dynamic thread structure |
| 7 | Capture system state to .blocs | Near must-have | Requires full metadata and pin offsets |
| 8 | Remove a block instance | Low value | Disconnect + ignore; requires item 5 |
| 9 | Add a block instance | Low value | Requires dynamic allocation + items 5, 6 |
| 10 | Add a new block definition | Not feasible | Requires recompilation |
| 11 | Add a new block spec | Not feasible | Requires recompilation |

Items 10 and 11 are impossible without recompilation. For items 8 and 9,
the preferred approach is to update the `.blocs` file and recompile rather
than attempting runtime installation.

## 9. Open Questions

- Hash algorithm selection (CRC32 vs SHA-1 vs other)
- Exact compressed BlockSpec encoding (escape sequences and keyword codes)
- Whether dummy signal addresses are ever needed for monitoring
- Whether the oscilloscope should be one block family or two separate blocks
- Protocol version numbering scheme
- Exact `HELLO` message format and field encoding
- Whether `GET_STRUCT_OFFSETS`, `GET_INSTANCE_ADDRS`, and `GET_SIGNAL_ADDRS`
  are separate commands or combined
- Minimum software receive buffer size recommendation
- Baud rate configuration mechanism
